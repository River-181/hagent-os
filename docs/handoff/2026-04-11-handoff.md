## 1. 현재까지 된 것

### 제품 구조

- 현재 제품은 `빈 껍데기`는 아니다.
- `local-first control plane`의 기본 뼈대는 이미 올라와 있다.
- 핵심 화면은 `dashboard / agents / cases / projects / skills / plugins / adapters / settings`까지 연결되어 있다.

### 온보딩

- `4-step onboarding wizard`가 있다.
- 온보딩에서 `Codex` 선택이 가능하다.
- 서버 `bootstrap`으로 아래를 한 번에 만든다.
  - `organization`
  - `starter agents`
  - `starter project`
  - `starter cases`
  - `installed skills`
  - `mounted skills`
  - `first dispatch`

### 에이전트 런타임

- 기본 adapter 구조가 있다.
  - `codex_local`
  - `claude_local`
  - `mock_local`
- 기본 agent type 실행 경로가 연결되어 있다.
  - `orchestrator`
  - `complaint`
  - `retention`
  - `scheduler`
- `orchestrator` 직접 wakeup 경로까지 수정 완료했다.
- `agent wakeup` route path 버그도 수정했다.

### 케이스 / 승인 / 활동

- `CaseDetail`에 `Approvals / Activity / Comments` 탭 구조가 들어갔다.
- approval payload와 scheduler side effect 저장 경로가 있다.
- activity log에 설정 변경, integration check, run, approval 이벤트가 쌓인다.
- 에이전트 결과는 케이스에 comment와 approval payload 형태로 남는다.

### 스킬

- filesystem-backed `skills registry`가 있다.
- 각 skill은 최소 다음 구조를 따른다.
  - `skill.package.json`
  - `SKILL.md`
  - `agents/openai.yaml`
- `SkillsPage`에서 아래를 볼 수 있다.
  - overview
  - files
  - `SKILL.md`
  - runtime
  - agents
  - source
- `AgentDetail`에서도 mounted skill preview가 보인다.

### 설정 / 관리자 화면

- `Settings`를 `paperclip` 참고 구조로 재구성했다.
- 현재 섹션:
  - `Company Settings`
  - `AI Policy`
  - `Integrations`
  - `Instance`
- `PATCH /api/organizations/:id` 추가 완료
- 저장 위치는 `organizations.agentTeamConfig`
- `AI Policy` 저장 시 기존 에이전트들의 `adapterType / adapterConfig`도 같이 갱신 가능

### Plugins / Adapters

- 페이지와 API는 있다.
- 현재 수준은 `registry + readiness manager`
- full SDK 단계는 아니다.

### 외부 연동

- `korean-law-mcp` 연결 경로가 있다.
- `Google Calendar` sync side effect 경로가 있다.
- 키가 없으면 degraded mode로 동작한다.
- 즉 `호출 경로 / readiness / side effect 저장`은 있지만, live verification은 아직 미완료다.

## 2. 지금 상태에서 꼭 알아야 할 현실

- 제품은 `paperclip 수준 완성형`은 아직 아니다.
- 지금은 `핵심 control plane 뼈대 + 일부 실제 동작`까지 올라온 상태다.
- 가장 중요한 건 `온보딩 이후 실제 업무가 자연스럽게 흐르는 경험`을 아직 완성하지 못했다는 점이다.

현재 부족한 부분:

- 사용자가 케이스를 승인하거나 보드에서 카드 상태를 바꿨을 때, 그 액션이 자연스럽게 다음 agent run으로 이어지는 흐름이 아직 약하다.
- 에이전트 결과가 `문서형 결과물`로 충분히 보기 좋게 정리되는 수준은 아직 아니다.
- `sub-case`나 후속 작업 chain 경험이 아직 약하다.
- `sidebar` 정보 구조가 아직 복잡하고 피로감이 있다.
- `ORG chart` 시각 품질과 구조 전달력이 아직 부족하다.
- `input channel ingestion`은 가장 중요한데, 실제 실험이 아직 부족하다.

## 3. 다음에 해야 할 것

### 1순위: 온보딩 후 첫 운영 경험 완성

온보딩이 끝난 직후 아래가 자연스럽게 이어져야 한다.

- 모델 선택
- 학원 온보딩 프로젝트 생성
- starter case 생성
- 첫 dispatch
- dashboard / projects / cases / agents에 반영
- 사용자가 “이미 AI 팀이 일하고 있다”는 느낌을 즉시 받기

### 2순위: CaseDetail를 진짜 작업 허브로 만들기

현재 CaseDetail는 뼈대는 있지만 아직 작업 허브 완성형은 아니다.

추가로 필요한 것:

- complaint / retention / scheduler 결과를 `문서형 아웃풋`으로 렌더
- 사용자가 결과물에 코멘트 달기
- 코멘트로 후속 지시
- `sub-case` 생성
- linked documents / linked approvals / linked runs 강화
- case 안에서 이어서 일을 시키는 경험 만들기

### 3순위: 보드 액션과 에이전트 실행 연결

사용자가 카드 상태를 바꾸거나 승인하면 그것이 실제 agent work로 이어져야 한다.

필요한 연결:

- `approve / reject / request revision`
- 보드에서 `todo -> in_progress -> in_review` 이동
- 이동/승인 이벤트가 run 생성으로 이어지게
- case activity와 run transcript가 자연스럽게 연결되게

### 4순위: 입력 채널 실검증

실제 운영에서는 채널 입력이 제일 중요하다.

검증 필요:

- 카카오 채널 inbound
- 카카오 봇/상담 메시지 -> case 생성
- 민원 / 상담 / 결제 / 출석 이벤트 -> case 생성
- 어떤 이벤트를 어느 case type으로 매핑할지 정리

중요:

- 카카오 채널과 봇은 준비됐다고 들었지만, 실제로 앱에 들어오는지는 아직 테스트가 필요하다.
- 이 부분은 제품 성패에 직접 연결된다.

### 5순위: Plugins / Adapters / Settings 연동 강화

- 지금은 settings 저장은 되지만 plugins/adapters 화면과 완전히 하나로 묶인 상태는 아니다.
- org별 preference와 readiness를 더 일관되게 보여줘야 한다.
- integration 설정 저장이 실제 runtime 해석까지 가는 단계는 아직 남아 있다.

### 6순위: Heartbeat / dispatch 정책 고도화

현재 heartbeat는 아직 단순 매칭 성격이 남아 있다.

필요:

- 어떤 case를 먼저 집을지
- 어떤 agent가 처리 가능한지
- 어떤 상태의 case는 재실행할지
- scheduler / complaint / retention 우선순위

즉, 정책 기반 배정으로 올려야 한다.

### 7순위: UI 정보 구조 정리

paperclip 대비 현재 아쉬운 점:

- 왼쪽 패널이 과밀하다.
- 카테고리 구분과 순서 정리가 더 필요하다.
- ORG chart가 보기 좋고 직관적인 수준까지는 아니다.

정리 방향:

- operator가 덜 피곤한 구조
- 카테고리별 분리 강화
- `company / work / agents / company ops` 같은 구획감 강화

## 4. 팀원이 바로 이해해야 할 핵심 제품 그림

우리가 만들려는 최종 경험은 대략 이렇다.

1. 사용자가 온보딩을 시작한다.
2. 모델을 고르고 학원 운영 프로젝트를 시작한다.
3. AI 팀이 starter case와 프로젝트를 잡고 일을 시작한다.
4. 사용자는 케이스 뷰와 프로젝트 뷰에서 결과물을 확인한다.
5. 사용자는 승인하거나, 보드에서 상태를 움직이거나, 코멘트로 다시 지시한다.
6. 에이전트는 그 케이스를 이어서 작업한다.
7. 결과물은 케이스 안에 문서처럼 누적된다.
8. 필요하면 sub-case를 파고 후속 작업을 이어간다.
9. 외부 채널에서 들어온 상담/민원/출석/결제 입력도 case로 유입된다.

즉, 목표는 단순 챗봇이 아니라 `학원 운영용 AI control plane`이다.

## 5. 지금 당장 체크해야 할 기술 포인트

- `Settings`
  - `PATCH /api/organizations/:id` 동작 확인
  - `agentTeamConfig` 저장 shape 확인
- `Agent wakeup`
  - `/api/agents/:id/wakeup` 동작 확인
- `Orchestrator`
  - 개별 wakeup 시 더 이상 `Unsupported agentType: orchestrator`가 나지 않는지 확인
- `CaseDetail`
  - approval / comments / activity 흐름 점검
- `input channel`
  - webhook / inbound path 실제 테스트

## 6. 현재 확인된 미완료 / 주의점

- 서버 전체 타입체크는 아직 기존 `TS6305`/implicit-any 이슈 때문에 깨진다.
- 이번 작업은 기능 경로 기준으로 확인했고, 저장소 전체 타입 정리는 별도 트랙이다.
- `korean-law-mcp`, `Google Calendar`는 live credential 기준 실사용 검증이 아직 남아 있다.
- settings 저장은 되지만 integration credential 자체를 DB에 넣어서 실제 runtime에서 쓰는 구조는 아직 아니다.

## 7. 추천 작업 순서

1. `온보딩 -> starter project -> starter cases -> first dispatch` UX 마감
2. `CaseDetail` 문서형 결과물 + 후속 코멘트 + sub-case
3. `보드 액션 -> agent run` 연결
4. `카카오 채널 / 상담 / 민원 / 결제 / 출석` 입력 테스트
5. `plugins / adapters / settings` 통합 정리
6. heartbeat 정책 개선
7. sidebar / org chart / dashboard 정보 구조 개선

## 8. 카톡 공유용 3줄

아래 문장을 그대로 카톡에 공유하면 된다.

1. `hagent-os는 지금 bootstrap, agents, skills, approvals, settings까지 기본 control plane은 살아 있고, 빈 화면 단계는 이미 넘겼습니다.`
2. `다음 핵심은 온보딩 이후 starter project/case가 바로 돌고, case detail에서 승인·코멘트·재지시·sub-case가 자연스럽게 이어지는 흐름을 완성하는 것입니다.`
3. `특히 카카오 채널/상담/민원/결제/출석 입력이 실제로 case로 들어오는지 검증이 가장 중요하고, paperclip급 완성도는 그 이후 UI 구조와 작업 흐름을 다듬는 단계입니다.`
